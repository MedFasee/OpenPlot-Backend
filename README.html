# openPlot – Backend (branch `feature/backend_medplot`) Este documento apresenta o estado atual do backend do projeto **openPlot-backend**. O foco desta branch consiste na integração do backend .NET ao ecossistema **MedPlot**, disponibilizando uma API REST destinada a: - consultar dados e metadados de medidas provenientes de historizadores (openHistorian/GSF etc.); - expor buscas e resultados de plotagem em formato compatível com openPlot Web e MedPlot Web; - fornecer utilitários de ingestão (GSF) e importação de configurações XML (MedPlot). --- ## 1. Visão geral da solução A solution `openplot.sln` contém, nesta branch, os seguintes projetos principais: ### `OpenPlot.Api` Backend HTTP baseado em Minimal API, responsável por: - cadastrar e consultar *search runs*; - recuperar séries temporais para plotagem (tensão, corrente, sequência positiva, entre outras); - integrar o modelo de dados e nomenclaturas utilizadas no MedPlot. ### `OpenPlot.Ingestor.Gsf` Serviço responsável por: - conectar-se ao stack GSF (openHistorian, SNAPDB); - ler medidas brutas; - normalizar dados segundo o modelo openPlot; - persistir amostras em tabelas `openplot.*` no PostgreSQL. ### `OpenPlot.XmlImporter` Ferramenta destinada a: - importar arquivos XML do legado MedPlot; - interpretar terminais, PMUs, canais e configurações; - converter os dados para o modelo atual (`pdc`, `pmu`, `signal` etc.). ### `Config` Projeto auxiliar que reúne: - configurações compartilhadas; - tipos utilitários e constantes; - componentes comuns entre os demais módulos. A implementação utiliza **C#** e **.NET**, conforme especificado no arquivo `.csproj`. --- ## 2. Arquitetura lógica do backend Apesar do uso de Minimal API, o código é estruturado em camadas funcionais: ### 2.1. API / Endpoints (OpenPlot.Api) - Ponto de entrada HTTP. - Mapeamento de rotas e recepção de requisições vindas de frontends. ### 2.2. Aplicação / Serviços Responsável pela lógica de negócio, incluindo: - cadastro e listagem de *search runs*; - criação de rótulos (labels); - montagem de séries temporais para plotagem. ### 2.3. Infraestrutura / Acesso a dados Inclui: - conexão com PostgreSQL via Dapper; - leitura de dados provenientes do GSF; - conversão para DTOs utilizados pela API. ### 2.4. Ferramentas auxiliares Inclui: - importação de XML (MedPlot); - ingestor GSF; - serviços de tempo e fuso horário. --- ## 3. Projeto `OpenPlot.Api` ### 3.1. Propósito Disponibilizar uma API REST para: - gerenciamento de *search runs*; - acesso a dados prontos para visualização; - compatibilidade com conceitos do MedPlot (terminais, fases, tipos de grandeza). ### 3.2. Estrutura típica #### Program / Startup Inclui: - registro de serviços (banco, tempo, rótulos, plotagem); - definição de rotas (ex.: `/buscas-realizadas`, endpoints de plotagem). #### Modelos / DTOs Incluem: - `SearchRunRow` e `SearchRunItem`; - DTOs de séries temporais com timestamps, valores e metadados. #### Acesso a dados Inclui: - consultas SQL a `openplot.search_runs` e tabelas de séries; - métodos para recuperar runs, sinais vinculados e dados para plotagem. #### Serviços de domínio Exemplos: - `SearchService`: opera sobre runs de busca; - `PlotService`: resolve sinais, consulta dados, aplica filtros e monta resposta; - `LabelService`: gera descrições textuais padronizadas; - `TimeService`: converte timestamps UTC → fuso horário do Brasil. --- ## 4. Projeto `OpenPlot.Ingestor.Gsf` ### 4.1. Propósito Operar a ingestão de dados provenientes do stack GSF/openHistorian. ### 4.2. Fluxo típico 1. Leitura das configurações (conexões, intervalos, sinais). 2. Conexão ao openHistorian/GSF. 3. Recuperação de amostras no intervalo especificado. 4. Normalização de nomes, unidades e tipificação. 5. Persistência no banco PostgreSQL via Dapper. --- ## 5. Projeto `OpenPlot.XmlImporter` ### 5.1. Propósito Permitir a importação de XML usado pelo MedPlot. ### 5.2. Responsabilidades Incluem: - leitura dos arquivos XML no formato MedPlot; - tradução para entidades openPlot (Terminal, PMU, Signal); - resolução de relacionamentos; - atualização das tabelas correspondentes no banco. --- ## 6. Integração com o MedPlot A branch atual possibilita: - exposição de funcionalidades antes restritas ao MedPlot Desktop; - consumo dos dados por frontends como openPlot Web; - preservação da compatibilidade com o modelo de dados legado. ### Fluxo operacional 1. A interface define os parâmetros da busca. 2. A API registra ou consulta o *search run*. 3. A API recupera dados ingeridos e metadados importados. 4. Todos os dados necessários são retornados no formato esperado pelo frontend. --- ## 7. DevOps A implementar
